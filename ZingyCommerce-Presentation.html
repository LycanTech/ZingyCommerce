Overview

What Is ZingyCommerce?

ZingyCommerce is a fully functional online shop split into 5 independent microservices, each responsible for one area of the business. The services communicate through a central API Gateway that also handles authentication, routing, and logging.

The entire platform is packaged in Docker containers and deployed to Azure Kubernetes Service (AKS) using a fully automated pipeline — push code, approve the deployment, and the new version is live.

Architecture

The 5 Microservices

🌐

Frontend

Single-page app built with vanilla HTML/CSS/JS and served by nginx. Handles the full shopping journey — browse, cart, checkout, orders — without any framework dependency.

Port 80
nginx
🔀

API Gateway

The single public entry point for all requests. Routes traffic to the correct microservice, validates JWT tokens on protected routes, and logs every request using morgan.

Port 3000
http-proxy-middleware
👤

User Service

Handles account registration, login, and profile retrieval. Passwords are hashed with bcrypt. Successful login returns a signed JWT token valid for 24 hours.

Port 3001
/users/*
📦

Product Service

Full CRUD product catalogue with search and category filtering. Pre-seeds 5 products on first boot. Exposes a stock-reduction endpoint used by the Order Service.

Port 3002
/products/*
🧾

Order Service

Places orders and tracks their status lifecycle (pending → confirmed → shipped → delivered). Calls the Product Service to atomically reserve stock before confirming any order.

Port 3003
/orders/* 🔐
Design Principles
▸Each service owns its own SQLite database — no shared data stores
▸Services communicate only via HTTP — loosely coupled
▸All inter-service URLs are environment variables — swappable
▸Every service exposes a /health endpoint for Kubernetes liveness probes
▸Orders protected by JWT — gateway rejects unauthenticated requests

🔑 Authentication Flow Browser → POST /users/login → User Service returns JWT → Browser stores token → Sends Authorization: Bearer <token> on every order request → Gateway validates JWT before forwarding to Order Service.
☸ In Kubernetes (AKS) Each service runs as a Deployment (2 replicas). An internal ClusterIP Service gives each deployment a stable DNS name — e.g. http://product-service:3002. Only the API Gateway and Frontend get a public LoadBalancer IP.
📡 Service-to-Service Call When a user places an order, the Order Service calls PATCH /products/:id/stock on the Product Service to atomically reduce stock before confirming the order. If stock is insufficient, a 409 is returned.

Azure Resources

🗂

Resource Group

A logical container that groups all related Azure resources together. Makes it easy to manage, monitor, and delete the entire project as one unit.

zingy-ecommerce-dev-rg
🐳

Azure Container Registry

A private Docker image registry. All 5 service images are built and pushed here during CI. AKS pulls images from ACR using a managed identity — no passwords needed.

ACR Basic SKU

Azure Kubernetes Service

Fully managed Kubernetes cluster. Azure handles the control plane (API server, etcd, scheduler). We manage the node pool — 2× Standard_B2s VMs with auto-scaling enabled.

AKS · 2 nodes · auto-scale

Collects container logs and Kubernetes control-plane events. Used to query logs, set alerts, and build monitoring dashboards in Azure Monitor.

PerGB2018 · 30-day retention
🔑

Role Assignment (AcrPull)

Grants the AKS kubelet identity the built-in AcrPull role on the Container Registry. This allows Kubernetes nodes to pull Docker images without a username or password.

Managed Identity · RBAC

Stores the Terraform state file remotely so the whole team shares the same infrastructure state. Prevents two people accidentally creating duplicate resources.

Infrastructure as Code

Terraform

File Structure
📄 main.tf
Azure provider config, resource group, and remote state backend pointing to Azure Blob Storage.
📄 variables.tf
All inputs: location, environment (dev/staging/prod), prefix, node count, VM size, ACR SKU.
📄 acr.tf
Container Registry resource + AcrPull role assignment so AKS can pull images without credentials.
📄 aks.tf
AKS cluster with system-assigned identity, auto-scaling node pool, Azure Monitor integration.
📄 monitoring.tf
Log Analytics workspace + AKS diagnostic settings forwarding control-plane logs.
📄 outputs.tf
Prints ACR login server, AKS cluster name, and a ready-to-paste az aks get-credentials command.
Deployment Workflow
1
terraform init — downloads Azure provider plugin, connects to remote state backend
2
terraform plan -out=tfplan — previews exactly what will be created/changed, saves plan
3
terraform apply tfplan — creates/updates real Azure resources from the saved plan
4
terraform output — prints ACR URL and AKS cluster name for use in next steps
# acr.tf — Container Registry + AKS pull permission resource "azurerm_container_registry" "main" { name = "${var.prefix}commerceacr${var.environment}" sku = var.acr_sku } resource "azurerm_role_assignment" "aks_acr_pull" { scope = azurerm_container_registry.main.id role_definition_name = "AcrPull" principal_id = azurerm_kubernetes_cluster .main.kubelet_identity[0].object_id }
💡 The state file in Azure Blob Storage means the whole team shares the same infrastructure view. Terraform will never create a duplicate resource if it already exists.

🔵

① App Pipeline

Triggers on services/**, frontend/**, helm/**
Stage 1 — Build (parallel matrix)
az acr login via service principal — no admin credentials needed
Build & push all 5 images simultaneously (matrix strategy)
Each image tagged with commit SHA + latest
Stage 2 — Deploy (GitOps trigger)
Update image.tag in helm/values.yaml with yq
Commit & push [skip ci] → ArgoCD detects → rolling update
→

② Infra Pipeline

Triggers on infrastructure/** only
Stage 1 — Plan (auto)
terraform fmt / validate / plan → artifact uploaded
Stage 2 — Apply ⏸ manual approval
Azure DevOps: Environments → production
GitHub Actions: Settings → Environments → production
terraform apply tfplan — creates ACR, AKS, Log Analytics
Managed OS disk (30 GB) — compatible with Standard_B2s
→

③ Bootstrap Pipeline

Manual trigger only — run once after AKS is created
az aks get-credentials — connects kubectl to cluster
Helm install: NGINX Ingress · cert-manager · ClusterIssuer
Helm install: Prometheus + Grafana · Loki
Install ArgoCD · apply appproject.yaml + application.yaml
ArgoCD takes over — no kubectl apply ever again
✅   Only the changed layer runs — app pushes never trigger Terraform, infra changes never rebuild images. Bootstrap runs once and is never needed again.

Containers & Orchestration

Docker & Kubernetes

🐳 Docker — Packaging
# Multi-stage Dockerfile (user-service) FROM node:20-alpine AS builder WORKDIR /app COPY package*.json ./ RUN npm install --omit=dev FROM node:20-alpine # final image WORKDIR /app COPY --from=builder /app/node_modules . COPY . . EXPOSE 3001 CMD ["node", "src/index.js"]
📋
docker compose up --build
Runs ALL 5 services locally with one command. Uses an internal bridge network so services reach each other by name.
🔒
Multi-stage build
Stage 1 compiles dependencies. Stage 2 copies only what's needed to run. Final image is smaller and more secure.
🌐
Frontend: nginx Alpine
Static HTML/CSS/JS served by nginx. No Node.js at runtime — very lightweight container.
☸ Kubernetes — Orchestration
📁
Namespace: zcommerce
All resources live in the same namespace, isolated from other apps running in the cluster.
🚀
Deployments × 2 replicas each
Each service runs 2 pods. If one crashes, the other serves traffic while Kubernetes restarts the failed pod.
🔌
Services (ClusterIP + LoadBalancer)
Internal services use ClusterIP (private). API Gateway and Frontend use LoadBalancer to get a public Azure IP.
💓
Liveness & Readiness Probes
Kubernetes polls /health every 30s. If unhealthy, the pod is restarted automatically.
📈
Horizontal Pod Autoscaler (HPA)
When CPU exceeds 70%, Kubernetes adds pods (up to 5 max). Scales back down when load drops to save cost.
🔐
Kubernetes Secrets
JWT signing key stored as a Kubernetes Secret, not hardcoded in the image or environment files.

Cloud Deployment — Test Users Ready

Prerequisites
☁ Azure Subscription 🔧 Azure CLI 🏗 Terraform ≥ 1.7 ☸ kubectl 🐳 Docker 🔑 az login
1
Create Terraform State Backend
az storage account create --name <NAME>tfstate --sku Standard_LRS
One-time setup. Stores shared Terraform state in Azure Blob Storage.
2
Configure terraform.tfvars
cp terraform.tfvars.example terraform.tfvars
Set your prefix, region (eastus), and storage account name. Never commit this file.
3
Provision Azure Infrastructure
terraform init && terraform plan -out=tfplan && terraform apply tfplan
Creates AKS cluster, ACR, Log Analytics workspace. Takes ~12 minutes.
4
Connect kubectl to AKS
az aks get-credentials --resource-group <RG> --name <AKS>
Downloads cluster credentials. Verify with: kubectl get nodes
5
Build & Push Docker Images to ACR
az acr build --registry <ACR> --image user-service:v1 ./services/user-service
Repeat for all 5 services. ACR builds in the cloud — no local Docker needed.
6
Create Namespace & Secrets
kubectl create ns zcommerce
kubectl create secret generic ecommerce-secrets --from-literal=jwt-secret="$(openssl rand -hex 32)" -n zcommerce
7
Deploy All Services to AKS
kubectl apply -f k8s/ -n zcommerce
Applies all Deployments, Services, and HPAs. Kubernetes pulls images from ACR automatically.
8
Wait for Pods to Become Ready
kubectl rollout status deploy/api-gateway -n zcommerce
Run for each service. All pods healthy = green light for test users.
9
Get Public IPs & Share with Testers
kubectl get svc -n zcommerce
Copy the EXTERNAL-IP for frontend (port 80) and api-gateway (port 80). Share the frontend IP with test users.
✅ Test User Journey
1. Open http://<FRONTEND-IP> in browser
2. Click Register → create account with email + password
3. Log in → browse products → add to cart
4. Place order → view order history
5. Test on mobile too — nginx serves static HTML to any device
💡 Cost tip: Run az aks stop --name <AKS> --resource-group <RG> after the presentation to pause the cluster and stop VM billing.

CI/CD Pipelines

Pipeline Deployment — 6 Steps in Order

⚠️ Complete these steps before running any pipeline. Skipping any step will cause the first run to fail.
1
Push Code to Both Remotes
git push -u github main
git push -u azure main
Both pipelines watch the main branch. Code must exist on both remotes before any trigger fires.
2
Create a Service Principal
az ad sp create-for-rbac --name "zcommerce-pipeline-sp" --role Contributor --sdk-auth
Save the full JSON — paste as AZURE_CREDENTIALS in GitHub Secrets and as the Azure DevOps service connection.
3
Bootstrap Infrastructure Manually (first time only)
cd infrastructure && terraform init
terraform apply -var="prefix=zcommerce" -var="environment=dev" -auto-approve
Stage 1 pushes images to ACR before Stage 2 creates it — ACR must exist first. Run once, then pipelines take over.
4
Set Up Azure DevOps Pipelines (3 pipelines)
New Pipeline → Existing YAML → app-pipeline.yml (repeat for infra-pipeline.yml, bootstrap-pipeline.yml)
Each pipeline needs: AZURE_SERVICE_CONNECTION, ACR_LOGIN_SERVER, TF_STORAGE_ACCOUNT, TF_RESOURCE_GROUP. No Docker Registry service connection — uses az acr login.
5
Set Up GitHub Actions Pipelines (3 workflows auto-registered)
Settings → Secrets → Actions → New repository secret
Add: AZURE_CREDENTIALS (full SP JSON), ACR_LOGIN_SERVER, TF_STORAGE_ACCOUNT, TF_RESOURCE_GROUP, GIT_USER_EMAIL, GIT_USER_NAME. No ACR_USERNAME/ACR_PASSWORD — uses az acr login. Create production environment with required reviewers.
6
Trigger Both Pipelines
git push azure main && git push github main
Each push triggers its pipeline independently. Stage 2 (Terraform Apply) requires manual approval before infrastructure changes are applied.
Pipeline Stages (both pipelines identical)
Stage 1 CI → Stage 2 Terraform Apply ✋ → Stage 3 Bootstrap K8s
✋ = manual approval gate before infra changes apply
